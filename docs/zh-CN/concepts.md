# 核心概念

## 什么是 Agent Loop？

**Agent Loop** 是一个调度的、有状态的工作流，AI 代理在其中：
1. **读取** 持久化状态（`STATE.md`）
2. **分类** 新信息（CI 失败、PRs、issues）
3. **行动** 在定义的边界内（报告 → 修复 → 合并）
4. **写入** 更新状态供下次运行

> "你不应该再手动提示编程代理了。你应该设计能够提示代理的循环系统。"

## Loops vs Goals

| 方面 | Loop | Goal |
|------|------|------|
| **目的** | 发现 + 分类持续工作 | 完成有界任务 |
| **节奏** | 调度（每 5 分钟、1 天等） | 运行直到完成 |
| **状态** | 持久化（`STATE.md`） | 临时（仅会话） |
| **停止条件** | 永不（手动终止开关） | 可验证的完成 |
| **示例** | "每天早上 9 点分类" | "修复所有失败的测试" |

**使用 loops：** 监控、分类、维护、基于节奏的工作
**使用 goals：** 有明确完成标准的有界任务

## 意图债务 vs 理解债务

### 意图债务

**定义：** 你告诉 loop 做什么和它实际做什么之间的差距。

**原因：**
- 模糊的提示（"修复 CI"）
- 缺少约束（没有拒绝列表）
- 没有验证步骤

**症状：**
- Loop 做出意外的更改
- Token 成本飙升
- 你害怕检查它做了什么

**解决方案：**
- 在 `LOOP.md` 中编写明确的约束
- 添加拒绝列表路径
- 需要验证者批准

### 理解债务

**定义：** loop 做了什么和你理解什么之间的差距。

**原因：**
- 不阅读 `STATE.md` 更新
- 跳过运行日志
- 复杂的多步操作

**症状：**
- "这个 loop 到底做了什么？"
- 对 PRs 感到惊讶
- 无法解释 token 花费

**解决方案：**
- 每次运行都阅读 `STATE.md`（第 1 周 +）
- 需要 5 行摘要
- 升级模糊的操作

## Harness 工程 vs Loop 工程

**Harness 工程** 构建工具、技能和基础设施，使 loops 成为可能。这是你构建一次然后重复使用的部分。

**Loop 工程** 设计实际运行的循环系统：提示、约束、验证和节奏。这是你为每个用例定制的部分。

### Harness 组件

- CLI 工具（`loop-audit`、`loop-init`、`loop-cost`）
- 技能库（`SKILL.md` 文件）
- MCP 服务器集成
- 工作树管理
- 状态文件格式

### Loop 组件

- 调度配置（cron、Automations）
- 约束定义（允许/禁止的操作）
- 验证规则（何时需要人工审查）
- 节奏和预算
- 升级策略

**关键区别：** Harness 是通用的，Loop 是特定的。不要将特定于项目的逻辑构建到 harness 中。

## 状态化记忆

Loops 需要跨会话的记忆。这是通过以下实现的：

### STATE.md

每个 loop 的主状态文件。包含：
- 上次运行时间
- 待处理项目
- 连续失败计数
- 当前模式（报告/修复/合并）

### LOOP.md

活动 loops 及其配置：
- 节奏（1d、5m 等）
- 模式（L1/L2/L3）
- 约束和拒绝列表
- 升级联系人

### 运行日志

每次 loop 执行的历史：
- 时间戳
- Token 花费
- 采取的操作
- 结果（成功/失败/升级）

## 安全等级

| 等级 | 名称 | 描述 | 使用场景 |
|------|------|------|----------|
| **L1** | 仅报告 | 只读分析，无更改 | 新 loops、高风险区域 |
| **L2** | 辅助修复 | 创建 PRs 供审查 | 大多数开发工作流 |
| **L3** | 自动合并 | 带门禁的自动合并 | 成熟的、经过验证的 loops |

**从 L1 开始**。仅在连续成功运行后升级到 L2/L3。
